## 業務規則分類

### 規則命名慣例

**格式**: `BR-[領域縮寫]-[編號]`

**範例**:
- `BR-ORD-001`: 訂單領域規則 001
- `BR-INV-001`: 庫存領域規則 001
- `BR-USR-001`: 使用者領域規則 001
- `BR-AC-001`: 存取控制領域規則 001

### 規則優先級

- **P0 (Critical)**: 必須遵守的核心業務規則,違反將導致嚴重業務錯誤
- **P1 (High)**: 重要業務規則,違反將影響業務正常運作
- **P2 (Medium)**: 一般業務規則,用於優化使用者體驗
- **P3 (Low)**: 可選的業務規則,未來可能調整

#### BR-ORD-001: 訂單金額計算規則

| 項目 | 內容 |
|------|------|
| **規則 ID** | BR-ORD-001 |
| **規則名稱** | 訂單金額計算規則 |
| **優先級** | P0 (Critical) |
| **狀態** | ✅ 已實作 / ⚠️ 待實作 / ❌ 已廢棄 |

**規則描述**:
```
訂單總金額 = 商品小計 - 折扣金額 + 運費 + 稅金

其中:
- 商品小計 = Σ(商品單價 × 數量)
- 折扣金額 = 優惠券折扣 + 會員折扣 + 活動折扣
- 運費 = 依據配送方式與地區計算
- 稅金 = (商品小計 - 折扣金額 + 運費) × 稅率
```

**業務邏輯**:
- 折扣金額不可大於商品小計
- 最終總金額不可為負數
- 計算結果四捨五入到小數點後兩位

**適用情境**:
- 購物車結帳
- 訂單建立
- 訂單金額驗證

**例外情況**:
- 若使用忠誠點數折抵,點數折抵額度計入折扣金額
- 企業客戶訂單可能享有零稅金優惠

**驗證方式**:

- [ ] BDD 場景: `features/order-checkout.feature::訂單金額計算`
- [ ] 手動測試案例: TC-ORD-001

**相關規則**:

#### BR-ORD-002: 運費計算規則

| 項目 | 內容 |
|------|------|
| **規則 ID** | BR-ORD-002 |
| **規則名稱** | 運費計算規則 |
| **優先級** | P0 (Critical) |
| **狀態** | ✅ 已實作 |

**規則描述**:
```
運費計算邏輯:
1. 訂單金額 ≥ 免運門檻 → 運費 = 0
2. 訂單金額 < 免運門檻 → 依據配送方式與地區查表

免運門檻:
- 一般會員: 1000 元
- VIP 會員: 500 元
- 企業客戶: 0 元 (免運)
```

**配送方式運費表**:

| 配送方式 | 本島 | 離島 | 國際 |
|---------|------|------|------|
| 標準配送 | 100 | 200 | 500 |
| 快速配送 | 150 | 300 | - |
| 超商取貨 | 60 | - | - |

**業務邏輯**:
- 運費僅在訂單建立時計算一次
- 訂單修改不影響已計算的運費
- 離島配送不提供快速配送選項
- 國際配送不提供超商取貨選項

**適用情境**:
- 購物車結帳
- 訂單金額計算

**例外情況**:
- 促銷期間可能有全站免運活動
- 特殊商品 (如大型家電) 運費另計

**驗證方式**:
- [ ] 單元測試: `test_shipping_fee_calculation()`
- [ ] BDD 場景: `features/order-checkout.feature::運費計算`

**相關規則**:

#### BR-ORD-003: 訂單取消規則

| 項目 | 內容 |
|------|------|
| **規則 ID** | BR-ORD-003 |
| **規則名稱** | 訂單取消規則 |
| **優先級** | P1 (High) |
| **狀態** | ✅ 已實作 |

**規則描述**:
```
訂單可取消的條件:
1. 訂單狀態 = "待付款" → 可直接取消
2. 訂單狀態 = "已付款" 且 商品未出貨 → 可取消 (需退款)
3. 訂單狀態 = "已出貨" → 不可取消 (需走退貨流程)
4. 訂單狀態 = "已完成" → 不可取消
```

**取消後動作**:
1. 更新訂單狀態為 "已取消"
2. 釋放預留庫存
3. 若已付款,啟動退款流程
4. 回收已使用的優惠券
5. 發送取消通知給客戶

**時間限制**:
- 訂單建立後 30 分鐘內未付款自動取消
- 付款後 24 小時內可由客戶取消

**適用角色**:
- 客戶: 可取消自己的訂單
- 客服: 可取消任何訂單 (需記錄原因)
- 系統: 自動取消逾時訂單

**業務邏輯**:
- 取消動作不可逆
- 退款方式必須與原付款方式一致
- 系統自動取消需發送 Email 通知

**驗證方式**:
- [ ] BDD 場景: `features/order-management.feature::訂單取消`
- [ ] 整合測試: `test_order_cancellation_workflow()`

**相關規則**:
- BR-INV-002: 庫存釋放規則
- BR-PAY-001: 退款處理規則

### [領域 2: 如庫存管理]

#### BR-INV-001: 可售庫存計算規則

| 項目 | 內容 |
|------|------|
| **規則 ID** | BR-INV-001 |
| **規則名稱** | 可售庫存計算規則 |
| **優先級** | P0 (Critical) |
| **狀態** | ✅ 已實作 |

**規則描述**:
```
可售庫存 = 實際庫存 - 預留庫存 - 安全庫存

其中:
- 實際庫存: 倉庫實際數量
- 預留庫存: 已被訂單預留但未出貨的數量
- 安全庫存: 為避免缺貨保留的最低數量
```

**業務邏輯**:
- 可售庫存不可為負數
- 當可售庫存 = 0 時,商品顯示 "暫時缺貨"
- 當可售庫存 < 預警值 時,發送補貨通知

**庫存狀態顯示**:
- 可售庫存 ≥ 10: 顯示 "現貨供應"
- 1 ≤ 可售庫存 < 10: 顯示 "庫存有限,僅剩 X 件"
- 可售庫存 = 0: 顯示 "暫時缺貨"

**適用情境**:
- 商品列表頁顯示
- 加入購物車驗證
- 訂單建立驗證

**驗證方式**:
- [ ] 單元測試: `test_available_inventory_calculation()`
- [ ] BDD 場景: `features/inventory-management.feature::庫存計算`

#### BR-INV-002: 庫存預留與釋放規則

| 項目 | 內容 |
|------|------|
| **規則 ID** | BR-INV-002 |
| **規則名稱** | 庫存預留與釋放規則 |
| **優先級** | P0 (Critical) |
| **狀態** | ✅ 已實作 |

**規則描述**:
```
庫存預留時機:
- 訂單建立時立即預留

庫存釋放時機:
1. 訂單取消 → 立即釋放
2. 訂單逾時未付款 (30分鐘) → 自動釋放
3. 訂單付款失敗 → 立即釋放

庫存扣減時機:
- 訂單出貨時,將預留庫存轉為實際扣減
```

**業務邏輯**:
- 預留庫存有時效性 (30 分鐘)
- 預留失敗則訂單無法建立
- 同一商品可被多個訂單預留,直到超過可售庫存

**併發處理**:
- 使用資料庫層級鎖 (pessimistic locking) 防止超賣
- 預留操作需為原子性 (atomic)

**驗證方式**:
- [ ] 併發測試: `test_concurrent_inventory_reservation()`
- [ ] BDD 場景: `features/inventory-management.feature::庫存預留`

### [領域 3: 如促銷活動]

#### BR-PRM-001: 優惠券折扣規則

| 項目 | 內容 |
|------|------|
| **規則 ID** | BR-PRM-001 |
| **規則名稱** | 優惠券折扣規則 |
| **優先級** | P1 (High) |
| **狀態** | ✅ 已實作 |

**規則描述**:
```
優惠券可用條件:
1. 優惠券狀態 = "有效"
2. 當前時間 ∈ [有效開始時間, 有效結束時間]
3. 優惠券使用次數 < 使用次數上限
4. 訂單金額 ≥ 最低消費金額
5. 使用者會員等級 ≥ 優惠券要求等級 (若有限制)

折扣計算:
- 百分比折扣: 折扣金額 = 訂單金額 × 折扣百分比
- 固定金額折扣: 折扣金額 = 優惠券面額
- 滿額折扣: 訂單金額 ≥ 門檻 → 折扣金額 = 優惠券面額
```

**疊加規則**:
- ✅ 優惠券 + 會員折扣: 可疊加
- ❌ 優惠券 + 優惠券: 不可疊加 (僅能使用一張)
- ✅ 優惠券 + 活動價: 可疊加
- ✅ 優惠券 + 忠誠點數: 可疊加

**使用後動作**:
- 增加優惠券使用次數
- 記錄使用歷史 (訂單 ID, 使用者 ID, 使用時間)
- 若達使用上限,更新狀態為 "已用完"

**退款處理**:
- 訂單取消或退款時,優惠券可重複使用

**驗證方式**:
- [ ] BDD 場景: `features/promotion-management.feature::優惠券使用`
- [ ] 單元測試: `test_coupon_discount_calculation()`

#### BR-MEM-001: 會員等級升降規則

| 項目 | 內容 |
|------|------|
| **規則 ID** | BR-MEM-001 |
| **規則名稱** | 會員等級升降規則 |
| **優先級** | P2 (Medium) |
| **狀態** | ⚠️ 待實作 |

**規則描述**:
```
會員等級計算基礎:
- 近 12 個月累計消費金額 (滾動計算)

等級門檻:
- Bronze (銅牌): $0 - $4,999
- Silver (銀牌): $5,000 - $14,999
- Gold (金牌): $15,000 - $49,999
- Platinum (白金): $50,000 以上

計算時機:
- 每月 1 號凌晨 00:00 自動重新計算所有會員等級
- 訂單完成時即時檢查是否達升級條件 (僅升不降)
```

**升級規則**:
- 達到門檻立即升級
- 發送升級通知 Email
- 記錄升級歷史

**降級規則**:
- 僅在每月 1 號批次計算時才會降級
- 降級前 7 天發送預警通知
- 記錄降級原因

**會員權益**:
| 等級 | 折扣 | 運費門檻 | 生日禮 | 專屬客服 |
|------|------|---------|--------|---------|
| Bronze | - | 1000 | - | - |
| Silver | 5% | 800 | ✓ | - |
| Gold | 8% | 500 | ✓ | ✓ |
| Platinum | 10% | 0 (免運) | ✓ | ✓ |

**驗證方式**:

#### BR-MEM-002: 忠誠點數規則

| 項目 | 內容 |
|------|------|
| **規則 ID** | BR-MEM-002 |
| **規則名稱** | 忠誠點數規則 |
| **優先級** | P2 (Medium) |
| **狀態** | ✅ 已實作 |

**規則描述**:
```
點數獲得:
- 消費 $100 獲得 1 點
- 訂單完成後立即發放
- 退貨訂單扣回對應點數

點數使用:
- 1 點 = $1 折抵
- 單筆訂單最多使用 30% 的訂單金額
- 使用點數後不影響會員等級累計消費

點數有效期:
- 點數自獲得日起 2 年內有效
```

**業務邏輯**:
- 點數餘額不可為負數
- 點數使用遵循 FIFO (先進先出) 原則
- 點數不可轉讓或兌換現金

### [領域 X: 存取控制]

#### BR-AC-001: 審核與建立互斥原則

| 項目 | 內容 |
|------|------|
| **規則 ID** | BR-AC-001 |
| **規則名稱** | 審核與建立互斥（Maker-Checker） |
| **優先級** | P0 (Critical) |
| **狀態** | ⚠️ 待實作 |
| **haARM 對應** | `constraint: approver_submitter_exclusion` |

**規則描述**:
```
建立者（Maker）不能同時擔任審核者（Checker）
→ haARM constraint: mutual_exclusion, not_both: [submitter, approver]

適用場景:
- 訂單審核: 建立訂單的人不能審核同一筆訂單
- 退款審核: 申請退款的客服不能審核同一筆退款
- 商品上架: 建立商品的人不能審核同一商品的上架申請
```

**業務邏輯**:
- 系統應檢查操作者身份，拒絕 Maker 對同一資源執行 Checker 操作
- 此規則為法規遵循要求，不可透過任何角色覆寫

**驗證方式**:
- haARM constraint 靜態檢查
- BDD 場景覆蓋（正向：不同人審核通過；反向：同一人審核被拒絕）

**相關規則**:
- BR-ORD-003: 訂單取消規則（取消操作也受此約束）

---

**填寫指南**:

1. **明確性**: 使用精確的數學表達式或邏輯運算式
2. **可測試性**: 每個規則都應該能轉換為可執行的測試
3. **可追溯性**: 明確標註規則與 BDD 場景、API、資料模型的對應關係
4. **可維護性**: 記錄變更歷史,追蹤規則演進
5. **完整性**: 考慮正常情況、異常情況、邊界條件

**最佳實踐**:
- 使用業務語言而非技術術語
- 提供具體範例幫助理解
- 規則應該是原子性的 (一個規則只描述一件事)
- 定期與業務專家驗證規則的準確性
