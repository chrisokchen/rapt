## 通用語言原則

### 什麼是通用語言 (Ubiquitous Language)?

> 通用語言是團隊中所有成員（包括業務專家、開發人員、測試人員、設計師）都理解並使用的**一致性術語**。這些術語應該：

- ✅ **業務導向**: 使用業務領域的語言,非技術術語
- ✅ **明確無歧義**: 每個術語只有一個明確的定義
- ✅ **一致使用**: 在程式碼、文件、對話中都使用相同術語
- ✅ **持續演進**: 隨著領域知識加深而更新

### 使用場景

通用語言應出現在:
- **BDD 場景 (Gherkin)**: Feature 與 Scenario 描述
- **程式碼**: 類別名稱、方法名稱、變數名稱
- **API 規格**: 端點名稱、資料模型
- **資料庫**: 表名、欄位名
- **角色/權限模型 (haARM)**: Actor 名稱、Role 名稱、Resource 名稱、Permission 描述
- **文件**: 需求文件、設計文件
- **日常對話**: 團隊會議、討論

## 詞彙表

### [領域 1: 如 訂單管理]

#### 訂單 (Order)

| 項目 | 內容 |
|------|------|
| **中文** | 訂單 |
| **英文** | Order |
| **複數** | Orders |
| **定義** | 客戶對商品的購買請求,包含一或多個商品項目、配送資訊與付款資訊 |
| **同義詞** | 購買單、購物單 (不建議使用) |
| **相關術語** | 訂單項目 (Order Item)、訂單狀態 (Order Status)、購物車 (Shopping Cart) |

**屬性**:
- 訂單編號 (Order Number): 唯一識別碼
- 訂單狀態 (Status): pending | paid | shipped | delivered | cancelled
- 小計 (Subtotal): 商品總金額
- 折扣金額 (Discount Amount): 優惠折扣
- 運費 (Shipping Fee): 配送費用
- 稅金 (Tax): 稅額
- 總金額 (Total): 最終應付金額

**使用情境**:
- 「客戶建立**訂單**後,系統會預留庫存」
- 「**訂單**狀態從 pending 變更為 paid」
- 「管理員可查看所有**訂單**列表」

**不變條件 (Invariants)**:
- 訂單總金額 = 小計 - 折扣金額 + 運費 + 稅金
- 已付款訂單不可直接取消(需走退款流程)
- 訂單必須至少有一個訂單項目

**程式碼範例**:
```typescript
class Order {
  id: string;
  orderNumber: string;
  status: OrderStatus;
  items: OrderItem[];
  subtotal: Money;
  discountAmount: Money;
  shippingFee: Money;
  tax: Money;
  total: Money;
}
```

#### 購物車 (Shopping Cart)

| 項目 | 內容 |
|------|------|
| **中文** | 購物車 |
| **英文** | Shopping Cart |
| **定義** | 客戶暫存的待購商品清單,尚未轉換為訂單 |
| **同義詞** | 購物籃 (Shopping Basket) - 英國用語 |
| **相關術語** | 購物車項目 (Cart Item)、訂單 (Order) |

**生命週期**:
1. 客戶將商品加入購物車
2. 客戶調整購物車內容(數量、移除)
3. 客戶進入結帳流程
4. 購物車轉換為訂單

**使用情境**:
- 「客戶將商品加入**購物車**」
- 「**購物車**中的商品數量可以隨時調整」
- 「結帳時,**購物車**內容會轉換為訂單」

**業務規則**:
- 購物車可以為空
- 購物車項目數量必須 > 0
- 購物車項目的庫存需即時檢查
- 遊客購物車與會員購物車需分別處理

#### 商品變體 (Product Variant)

| 項目 | 內容 |
|------|------|
| **中文** | 商品變體 |
| **英文** | Product Variant |
| **定義** | 同一商品的不同規格組合(如顏色、尺寸、容量) |
| **相關術語** | 商品 (Product)、SKU、庫存 (Inventory) |

**範例**:
- 商品: iPhone 15 Pro
  - 變體 1: 黑色 / 128GB (SKU: IP15P-BK-128)
  - 變體 2: 黑色 / 256GB (SKU: IP15P-BK-256)

**使用情境**:
- 「每個**商品變體**都有獨立的 SKU 與價格」
- 「庫存以**商品變體**為單位管理」
- 「客戶選擇規格後,確定購買的**商品變體**」

### [領域 2: 如 庫存管理]

#### 庫存 (Inventory)

| 項目 | 內容 |
|------|------|
| **中文** | 庫存 |
| **英文** | Inventory |
| **定義** | 特定商品變體在特定倉庫的可用數量 |
| **相關術語** | 實際庫存、預留庫存、可售庫存、安全庫存 |

**庫存類型**:
- **實際庫存 (Actual Quantity)**: 倉庫實際持有的數量
- **預留庫存 (Reserved Quantity)**: 已被訂單預留但未出貨的數量
- **可售庫存 (Available Quantity)**: 可供銷售的數量 = 實際庫存 - 預留庫存
- **安全庫存 (Safety Stock)**: 為避免缺貨保留的最低數量

**計算公式**:
```
可售庫存 = 實際庫存 - 預留庫存 - 安全庫存
```

**使用情境**:
- 「**庫存**不足時,商品顯示為缺貨」
- 「訂單建立時立即預留**庫存**」
- 「當**可售庫存**低於安全庫存時發送補貨通知」

#### SKU (Stock Keeping Unit)

| 項目 | 內容 |
|------|------|
| **中文** | 庫存單位代碼 |
| **英文** | Stock Keeping Unit (SKU) |
| **定義** | 唯一識別每個商品變體的代碼 |
| **格式範例** | IP15P-BK-128 (商品代碼-顏色-容量) |

**命名規則**:
- 必須唯一
- 建議包含商品資訊(便於識別)
- 避免使用特殊字元
- 建議使用大寫字母與數字

**使用情境**:
- 「每個**SKU**對應唯一的商品變體」
- 「庫存管理以**SKU**為基礎單位」
- 「訂單明細記錄購買的**SKU**」

### [領域 3: 如 促銷活動]

#### 優惠券 (Coupon)

| 項目 | 內容 |
|------|------|
| **中文** | 優惠券 |
| **英文** | Coupon |
| **定義** | 可兌換折扣的代碼,有使用條件與期限 |
| **相關術語** | 促銷活動 (Campaign)、折扣 (Discount) |

**優惠券類型**:
- **百分比折扣**: 訂單金額打折(如 9折、85折)
- **固定金額折扣**: 減免固定金額(如折抵 $100)
- **滿額折扣**: 訂單金額達門檻才生效(如滿 $1000 折 $100)
- **買 X 送 Y**: 購買指定數量贈送商品

**使用條件**:
- 有效期限
- 使用次數限制
- 最低消費金額
- 會員等級限制

**使用情境**:
- 「客戶輸入**優惠券**代碼以獲得折扣」
- 「**優惠券**達使用上限後自動失效」
- 「每筆訂單僅能使用一張**優惠券**」

#### 會員等級 (Member Level)

| 項目 | 內容 |
|------|------|
| **中文** | 會員等級 |
| **英文** | Member Level / Membership Tier |
| **定義** | 根據消費金額或頻率劃分的客戶等級,享有不同權益 |
| **級別** | Bronze (銅牌) → Silver (銀牌) → Gold (金牌) → Platinum (白金) |

**升級條件**:
- 近 12 個月累計消費達門檻

**會員權益**:
| 等級 | 折扣 | 免運門檻 | 生日禮 | 專屬客服 |
|------|------|---------|--------|---------|
| Bronze | - | $1000 | - | - |
| Silver | 5% | $800 | ✓ | - |
| Gold | 8% | $500 | ✓ | ✓ |
| Platinum | 10% | 免運 | ✓ | ✓ |

**使用情境**:
- 「**會員等級**每月 1 號自動更新」
- 「Gold **會員**享有 8% 折扣」
- 「部分優惠券限 Platinum **會員**使用」

## 反模式 (Anti-patterns)

### ❌ 避免使用的術語

| 錯誤術語 | 問題 | 正確術語 | 說明 |
|---------|------|---------|------|
| Record | 太技術化 | Order / Product | 使用業務術語 |
| Data | 太抽象 | Customer / Product | 明確指出資料類型 |
| Item | 太籠統 | Order Item / Cart Item | 加上限定詞 |
| Info | 太模糊 | Customer Details / Product Specifications | 具體說明資訊類型 |
| Status Code | 技術術語 | Order Status / Payment Status | 使用業務語言 |

### ❌ 避免的命名模式

- **動詞 + 名詞的技術術語**: GetOrder → 應使用 Order
- **縮寫**: Prd, Ord, Inv → 應使用完整單字
- **模糊的修飾詞**: MainProduct, TempCart → 應明確說明

## 維護指南

### 如何新增術語

1. **識別術語來源**: Event Storming、利害關係人訪談、領域專家討論
2. **定義術語**: 與領域專家確認定義
3. **記錄關聯**: 關聯到其他術語、業務規則
4. **更新程式碼**: 確保程式碼中也使用相同術語
5. **通知團隊**: 在會議中宣布新術語

### 如何修改術語

1. **討論必要性**: 與團隊討論為何要修改
2. **影響評估**: 評估對程式碼、文件的影響
3. **制定遷移計畫**: 如何從舊術語過渡到新術語
4. **記錄在演進記錄中**: 保留歷史軌跡

---

**最佳實踐**:

1. **與領域專家共創**: 詞彙表應由業務專家與技術團隊共同建立
2. **保持活力**: 定期審查與更新,隨領域知識增長演進
3. **一致性至上**: 寧可用稍微不完美但一致的術語,也不要多種說法並存
4. **程式碼反映**: 程式碼應該如實反映通用語言,避免翻譯
5. **教育新成員**: 新成員加入時,詞彙表是重要的學習資源
