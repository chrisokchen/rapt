# RAscore Rubric v0.1

本文件是 `rapt-RAscore` 的評分規則 SSOT。每個準則以 0-3 分評分：

```text
0 = 缺失 / 失敗
1 = 部分達成，但有重大缺口
2 = 大致良好，僅有小缺口
3 = 完整 / 優異
```

總分公式：

```text
RAscore = 100 * Σ(維度平均分 / 3 * 維度權重)
```

## 維度與權重

| 維度 | 權重 | 準則 |
|---|---:|---|
| A. 需求覆蓋與保真度 | 20% | A1-A3 |
| B. Gherkin 行為規格品質 | 18% | B1-B5 |
| C. DBML 領域模型品質 | 14% | C1-C4 |
| D. Gherkin ↔ DBML 跨規格一致性 | 24% | D1-D4 |
| E. 追溯性與決策記錄 | 14% | E1-E3 |
| F. 可驗證與生成準備度 | 7% | F1-F2 |
| G. 流程穩定性 | 3% | G1-G2 |

## A. 需求覆蓋與保真度

### A1 來源需求覆蓋

- 0：大量來源需求未出現在 Gherkin 或 DBML。
- 1：只覆蓋主要 happy path，例外、邊界、權限或狀態規則大量缺漏。
- 2：大多數需求已覆蓋，少數次要需求缺漏或待確認。
- 3：來源需求幾乎全覆蓋，未納入者有明確排除理由。

### A2 Scope loss 控制

- 0：轉換過程中明顯流失需求且無記錄。
- 1：有多處語意流失，但未被標記。
- 2：少量流失，已記錄為決策、風險或待確認。
- 3：無未預期流失；刪減、合併或延後皆有原因。

### A3 Scope creep / Gold-plating 控制

- 0：大量新增來源需求未提及的功能、實體或規則。
- 1：有明顯腦補內容，未標記假設。
- 2：少量衍生內容，已標記為假設或待確認。
- 3：所有新增內容皆可回溯來源，或明確標為團隊假設。

## B. Gherkin 行為規格品質

### B1 業務導向與抽象層級

- 0：Scenario 充斥 UI 操作、按鈕、欄位、API 細節。
- 1：多數仍像操作腳本，業務人員難以直接確認。
- 2：大致使用業務語言，少數步驟仍需提升抽象層級。
- 3：完全以業務意圖、業務事件與可觀察結果描述。

### B2 Scenario 原子性

- 0：多條規則串成長流程，失敗原因不可診斷。
- 1：多數 Scenario 混合多個行為或規則。
- 2：多數 Scenario 聚焦單一行為，少數仍偏大。
- 3：每個 Scenario 驗證一條主要業務規則或狀態轉換。

### B3 Given / When / Then 結構

- 0：缺前置狀態、觸發行為或可觀察結果。
- 1：G/W/T 角色常混淆，Then 不可判定。
- 2：結構大致正確，少數步驟角色不清。
- 3：Given 是前置狀態，When 是單一觸發，Then 是可驗證結果。

### B4 業務規則、例外與邊界覆蓋

- 0：只有 happy path。
- 1：少量例外，核心邊界與否定路徑缺漏。
- 2：主要規則、例外與邊界大致覆蓋。
- 3：正常、例外、邊界、權限、狀態轉換都有系統性覆蓋。

### B5 統一語言一致性

- 0：同一概念多種說法，詞彙混亂。
- 1：部分詞彙與 glossary / DBML 不一致。
- 2：大致一致，少數同義詞需清理。
- 3：全面採用統一語言，並能對應 glossary / DBML。

## C. DBML 領域模型品質

### C1 實體與概念完整性

- 0：缺漏多個核心業務實體。
- 1：有主要實體，但重要關聯或屬性缺漏。
- 2：概念大致齊全，少數次要概念待補。
- 3：需求與 Gherkin 中的關鍵領域概念皆有歸宿。

### C2 關聯、基數與鍵

- 0：缺 PK/FK 或關聯，cardinality 不明。
- 1：部分關聯方向或基數錯誤。
- 2：關聯大致正確，少數需業務確認。
- 3：PK/FK、1:1、1:N、M:N 與中介表皆正確反映業務規則。

### C3 約束與狀態建模

- 0：約束缺失，狀態變化無欄位或 enum 支撐。
- 1：有部分約束，但必填、唯一、狀態值等不完整。
- 2：約束大致完整，少數狀態或邊界限制待補。
- 3：型別、nullable、unique、enum/status、時間欄位等清楚支撐業務規則。

### C4 命名與正規化

- 0：命名技術化或混亂，模型冗餘嚴重。
- 1：部分命名可讀，但仍有縮寫、雜訊或職責混合。
- 2：命名大致反映領域，正規化大致合理。
- 3：命名忠實反映統一語言；達 3NF 或去正規化有明確理由。

## D. Gherkin ↔ DBML 跨規格一致性

### D1 詞彙對齊

- 0：Gherkin 與 DBML 像在描述兩套不同系統。
- 1：部分名詞可對應，但大量同概念異名。
- 2：多數可對應，少數需 glossary 或人工橋接。
- 3：Gherkin 名詞與 DBML 實體 / 屬性高度一致。

### D2 行為 → 資料覆蓋

- 0：大量 Scenario 引用 DBML 無法支撐的概念、狀態或結果。
- 1：明顯比例的行為缺資料支撐。
- 2：多數行為有資料支撐，少數缺口。
- 3：每個 Scenario 涉及的核心概念、狀態與結果都能在 DBML 找到支撐。

### D3 資料 → 行為覆蓋

- 0：大量 DBML 實體沒有任何 Scenario 觸及。
- 1：明顯比例實體或關聯是孤兒。
- 2：多數實體被行為使用，少數有保留理由待補。
- 3：每個實體至少被一個 Scenario 觸及，或有明確保留理由。

### D4 狀態轉換與約束一致

- 0：Gherkin 的狀態與 DBML enum/status/constraint 明顯矛盾。
- 1：多個狀態或約束無法互相驗證。
- 2：大多數狀態與約束一致，少數值域需補。
- 3：Scenario 中的狀態轉換、限制條件與 DBML 狀態 / 約束完整對齊。

## E. 追溯性與決策記錄

### E1 前向追溯：需求 → 產出

- 0：無法從原始需求追到 Feature / Scenario / Entity。
- 1：僅部分需求可追。
- 2：多數需求可追到產出，少數斷鏈。
- 3：每條需求皆可追到對應 Scenario 與 DBML 概念。

### E2 後向追溯：產出 → 需求

- 0：Scenario 或 Entity 幾乎沒有來源依據。
- 1：僅部分產出可追回來源。
- 2：多數產出可追回來源，少數需補 trace。
- 3：每個 Scenario 與核心 Entity 都可追回來源需求或決策。

### E3 決策、假設、缺口記錄

- 0：沒有記錄假設、未決問題或建模決策。
- 1：記錄零散，無法支持審查。
- 2：主要決策與缺口有記錄，但格式不穩。
- 3：決策、假設、風險、排除項皆結構化記錄，可供後續追蹤。

## F. 可驗證與生成準備度

### F1 可驗證性

- 0：Then 多為主觀描述，無法判定通過 / 失敗。
- 1：部分結果可驗證，但仍有模糊語句。
- 2：多數 Scenario 可轉成驗收測試。
- 3：Scenario 的結果可明確斷言，具備測試生成基礎。

### F2 後續 DSL / Codegen readiness

- 0：不足以推導 haAPI / haPDL / haARM / 測試。
- 1：可推導少量產物，但大量需要人工補洞。
- 2：大多數場景與模型可支援後續規格推導。
- 3：足以作為後續 haAPI、haPDL、haARM、測試與 codegen 的輸入。

## G. 流程穩定性

### G1 評審一致性

- 0：不同評審分數與缺口判斷差異極大。
- 1：核心結論不穩，需要大量協調。
- 2：大方向一致，細節需校準。
- 3：不同評審對主要缺口與分數高度一致。

### G2 流程可重現性

- 0：沒有固定步驟與中間產物。
- 1：部分階段跳過或產物缺失。
- 2：流程大致可重現，但格式仍需統一。
- 3：提取、分解、分析、建模、評審皆有固定輸入輸出。

## 否決條件

以下任一成立，仍只作 advisory 標示，不阻擋流程：

1. `A1 = 0`
2. `D2 = 0`
3. `D3 = 0`
4. `E1 = 0 且 E2 = 0`
5. `B1 = 0`

若觸發任一否決條件，最高 advisory grade 為 C；若觸發兩項以上，advisory grade 為 D。

## 等級

| 分數 | 等級 | 判讀 |
|---:|---|---|
| 85-100 | A | 可作為 SSOT，允許進入後續推導 |
| 70-84 | B | 大致可用，需修正局部缺口 |
| 50-69 | C | 有重大缺口，需返工後重評 |
| < 50 | D | 不應作為後續規格來源 |
